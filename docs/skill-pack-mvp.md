# 技能库与技能包（Skill Pack）MVP

> 本文由 `docs/skill-preset-mvp.md` 按 grilling 共识改写。  
> 对外产品名：**技能包 / Skill Pack**。不使用裸词 **Preset**，以免与现有 **Tool Preset**（none / default / full）混淆。

## 1. 目标与 JTBD

**一句话：** 以前每个新项目都要重新搜/装技能；现在选一个**技能包**，相关技能一次装齐。

为 Pi Web 增加可复用的项目技能配置：

1. 用户把技能安装到一个指定的**本地技能库**。
2. 在库技能上创建**技能包**（场景合集）。
3. 把一个或多个技能包**应用到**任意工作空间（复制缺失技能到项目）。

### 原则

- **隐喻**：App Store 合集——技能包 = 精选安装合集；应用 = 安装缺失技能。不是素材库双写，不是配置档案切换。
- **项目写盘路径**与现网一致：项目技能位于 `<cwd>/.pi/skills/`（与 `lib/skill-lock.ts`、Skills 安装 UI 一致）。pi 另会发现 `.agents/skills`、全局 `~/.pi/agent/skills`、package skills 等；**本 MVP 只写入** `<cwd>/.pi/skills/`，不改其它来源。
- **技能库**是用户配置的一个本地目录；库内技能放在 `<libraryRoot>/.pi/skills/`，以便 `npx skills add` 以库目录为 cwd 时与项目安装行为一致。
- **应用技能包绝不覆盖**已存在的同名项目技能。
- **移除技能包只摘标签**，不自动删除项目内任何技能文件。
- 一个项目可以挂多个技能包；安装时取所有已选包的技能**去重并集**。
- 编辑技能包或更新库技能，**不会**自动修改已经应用过的项目。
- 库更新不静默改包引用；用户须在包编辑页显式「更新引用」后，**之后的新应用**才用新版本。

## 2. 术语

| 名称 | 定义 |
| --- | --- |
| 技能库 | 用户配置的可读写本地目录，保存可复用的技能原件。可以是用户自己维护的 Git 工作区；MVP **不负责** Git push/pull。 |
| 库技能 | `<libraryRoot>/.pi/skills/<skillKey>/` 中含 `SKILL.md` 的完整目录。 |
| 技能包（Skill Pack） | 有名称、说明和一组库技能版本引用的场景合集，例如「PPT大师」。 |
| 项目技能 | `<cwd>/.pi/skills/<skillKey>/` 中的技能。 |
| 冲突 | 项目目标路径已有同名技能；应用时**跳过**该技能，不覆盖、不改写。 |
| 有跳过 | 技能包已挂到项目，但至少一项技能因同名未写入。UI 用自然语言与徽章表示，**不用** `2/3` 分数。 |

`skillKey` 是技能所在目录名，比较时按大小写不敏感的规范化值处理。界面显示名称仍取 `SKILL.md` frontmatter 或资源加载器提供的名称。

**禁止对外使用：** Preset（技能语境）、托管技能、自动卸载、推送/拉取同步等包管理器话术（见 §6 明确不做项）。

## 3. MVP 范围

### 包含

- 配置一个默认本地技能库目录。
- 在技能库中浏览、查看详情、删除库技能。
- 从市场、本地目录和 Git 仓库将技能导入**技能库**（不直接写入当前项目）。
- 创建、编辑、删除技能包；包内技能仅从技能库选择。
- **主路径：** 在当前工作空间添加或移除技能包标签（预览 → 确认 → 写入）。
- **次要：** 若存在选择/创建 cwd 的流程，可挂上可选技能包；不是闭环门闩。
- 应用前预检；存在跳过时展示确认说明（将安装 / 因同名跳过）。

### 不包含

- 多个技能库、库之间同步、远端 Git push/pull。
- 从 zip、`.skill` 或批量目录导入；本地安装仅支持选择一个含 `SKILL.md` 的目录。
- 复杂文本或文件级合并。
- 技能包的团队分享、权限、云同步和导入导出。
- 后台自动同步或自动更新已应用项目。
- 与 Tool Preset（工具集 none/default/full）的任何合并或改名。

> **关于卸载与库对照：** §12 提供了卸载（带安装回执）、项目↔库覆盖、分叉另存等进阶能力的产品设计。它们不阻塞 MVP 核心闭环，属于 v1.1 / v1.2 分层交付的内容，但本文档一并给出完整蓝图，避免实现走样。

## 4. 信息架构与界面

### 4.1 Skills 模态框

现有 Skills 入口保留，一级结构建议：

1. **当前工作空间**  
   - 保留现有技能列表、详情、启用开关和原有安装能力。  
   - 顶部新增「已应用技能包」标签区（见 §4.4）。  
   - **不**展示与技能库的推拉同步状态（本 MVP 不做同步）。

2. **技能库**  
   - 当前库目录、修改库目录、库技能列表。  
   - 未配置目录时，主操作：「选择技能库目录」。

3. **获取技能**  
   - 二级：市场 / 本地 / Git。  
   - **安装目标固定为技能库**，不在此处向当前项目写文件。

原有「Add Skill」在「当前工作空间」保留（直接装到项目的现有行为不变），并可增加「从技能库添加」作为捷径（单技能复制，规则同应用：不覆盖同名）。

### 4.2 获取技能

**市场：** 热门列表、关键词搜索、安装。现有 `npx skills add` 以**技能库目录**为工作目录执行，不使用全局安装参数；完成后校验技能进入 `<libraryRoot>/.pi/skills`。MVP 只需可用的热门排序与搜索；Trending / Hot / 来源筛选 / 分页 / 取消安装可后置。

**本地：** 选择或填写含 `SKILL.md` 的目录，展示名称与说明后导入库。浏览器无法弹目录选择器时允许绝对路径；桌面复用已有目录选择。

**Git：** 输入 Git URL → 克隆到临时目录 → 扫描 `SKILL.md` → 用户勾选后写入库。临时目录仅用于预览与导入。

同名库技能不静默覆盖：显示「已存在」，用户选择取消、更新库技能或改名后导入。仍被某个技能包引用的库技能不能直接删除；列出引用它的技能包，要求先更新或删除引用。

### 4.3 技能包模态框

左侧导航新增 **技能包** 入口（不依赖是否选中工作空间），打开全局技能包模态框。

- 左侧：技能包列表（名称、技能数量）；底部 `+ 新建技能包`。
- 右侧：名称、说明；已选库技能（名称、简述、来源、版本短哈希）。
- 「添加技能」打开技能库多选。
- 删除技能包只删包定义，不删库内容，不改任何已应用项目。

技能包保存库技能的 `skillKey` + 内容哈希。MVP **不**保存库技能历史快照：当库中当前哈希与包引用不一致时，该引用显示「需要更新」，不能静默用新内容去应用。用户须显式：更新引用到当前版本、替换技能、或从包中移除。已挂到项目的技能文件不受影响。

### 4.4 工作空间中的技能包标签

「当前工作空间」Tab 顶部：

- `+`：从技能包列表多选并应用（先预览再确认）。
- 标签关闭：仅移除该包标签（见 §5.4），无清理弹窗、不删文件。
- 若应用时有同名跳过：标签旁徽章 **「有跳过」**；点击查看被跳过列表。  
  **禁止** `2/3 已应用` 式分数。

## 5. 应用技能包的规则

### 5.1 主路径：当前工作空间

用户选择目标技能包 ID 集合（相对当前已挂标签的目标状态）→ 预检 → 必要时确认 → 执行。

预检读取 `<cwd>/.pi/skills`，按所有目标技能包的技能并集生成计划：

| 情况 | 行为 |
| --- | --- |
| 无同名冲突 | 复制缺失技能；可不再弹确认（或轻量 toast）。 |
| 有同名冲突 | 确认框分组：**将安装** / **因同名跳过**；用户确认后继续。 |
| 包引用的库技能缺失，或内容哈希与包记录不匹配 | **阻止应用**，提示先在技能包中更新、替换或移除该技能。 |
| 单次复制失败 | 不写入/不更新本次应用状态中对应变更；新内容先入临时目录再原子移入；失败时回滚本次新建目录，避免半截技能目录。 |

已有项目中的其它技能不会因为新挂技能包被删除或覆盖。

同名已存在时：

- 不覆盖，不改写该目录。
- 预览标记「因同名跳过」。
- 技能包标签仍可挂上，并标记「有跳过」。

### 5.2 次要路径：选择 / 创建工作空间时

若产品已有「选择目录进入工作空间」流程，可在其后增加可选的技能包多选（可跳过），预检与执行逻辑与 §5.1 相同。
**验收与开发优先级以 §5.1 为准**；本路径不阻塞 MVP 闭环。

### 5.3 多技能包并集（仅安装）

目标标签集合变化时：

1. 读取所有目标技能包的库技能引用。
2. 以 `skillKey + contentHash` 去重，得到期望安装集合。
3. 同一 `skillKey` 对应不同 contentHash → **技能包间版本冲突**，不能自动选版本；阻止并提示用户先统一各包中的引用。
4. 期望集合中项目尚不存在的 → 生成安装（复制）；已存在同名 → 跳过。
5. **不**根据「不再被任何包引用」生成删除操作。

### 5.4 移除标签 vs 卸载技能包

这是两个不同的动作：

| 动作 | UI 入口 | 是否动文件 |
| --- | --- | --- |
| **移除标签** | 标签 × 按钮（快捷） | 否，只去标签 + 丢回执 |
| **卸载技能包…** | 标签右键或详情菜单 | 是，但用户确认后逐个勾选删除 |

**移除标签**（MVP 核心）：只从 `appliedPacks[]` 中去掉该项，`.pi/skills/` 下文件全部不动。技能列表中不展示长期「来自：某包」来源标注。

**卸载技能包**（v1.1）：依赖安装回执（见 §7.2、§12）——应用成功时记录 `{skillKey, contentHash}`。卸载时按回执现算：

- 回执技能仍在且哈希 = 回执记录 → 默认勾选「删除」
- 回执技能仍在但哈希已变 → 默认勾选「保留（已修改）」
- 同名技能也出现在其它已挂包的回执中 → 默认勾选「保留（被其它技能包共用）」

用户确认后才删；关闭标签不触发卸载。详见 §12.

## 6. 技能详情中的库对照

本节描述项目技能和库技能之间**一次性的、不用同步状态机**的对照操作。与旧稿不同：这里没有「已关联 / 基线 / 分叉状态机」常驻 UI；所有判断依据是当前项目目录哈希和库目录哈希的现算对比。

### 6.1 提示（现算，不是状态枚举）

在项目技能详情页，若存在同名库技能，现算并显示：

| 条件 | 提示 |
| --- | --- |
| 项目哈希 = 库哈希 | 「与库相同」— 无操作 |
| 项目哈希 ≠ 库哈希 | 「与库不同」— 可覆盖 / 另存 |

**不存在**「项目较新」「库较新」「已分叉」等状态标签。用户不需要知道谁改得更早，只需知道「一样还是不一样」。

若仅项目有、库没有同名技能 → 可显示「仅在本地项目」，并提供「另存到技能库」入口。

### 6.2 覆盖与另存（两个锤子）

在「与库不同」或「仅在本地项目」状态下提供：

| 操作 | 行为 | 安全措施 |
| --- | --- | --- |
| **用库版本覆盖项目** | 整目录原子替换 | 确认弹窗，说明将覆盖的内容 |
| **用项目版本更新库** | 整目录原子替换库中的副本 | 确认弹窗；不自动改技能包引用 |
| **另存为新库技能…** | 改名 skillKey 写入库 | 需输入新 skillKey；原库技能不动 |

- 覆盖操作：新内容先入临时目录，成功后再原子替换。
- 「用项目版本更新库」成功后：**不**自动更新任何技能包引用。若该 skillKey 被某技能包引用，技能包详情页继续显示「引用需更新」，用户需在包编辑页单独操作。
- 「另存为新库技能…」可以升级为新建技能包时的快捷动作。

### 6.3 不在 MVP 的

- 不维护长期同步关联。
- 不设基线哈希 / 共同祖先。
- 不对覆盖/另存做批量操作。

## 7. 持久化模型

### 7.1 全局（Pi Web 用户配置）

```json
{
  "version": 1,
  "libraryRoot": "/absolute/path/to-skill-library",
  "packs": [
    {
      "id": "uuid",
      "name": "PPT大师",
      "description": "演示文稿制作与审阅",
      "skills": [
        { "skillKey": "slides", "contentHash": "sha256..." }
      ]
    }
  ]
}
```

字段名使用 `packs` / 技能包语义；实现层可映射文件名，但 API 与 UI 文案不用 preset。

### 7.2 项目状态

路径建议：`<cwd>/.pi/skill-packs.json`（避免再引入易误解的 preset 字样）。

```json
{
  "version": 1,
  "appliedPacks": [
    {
      "packId": "uuid",
      "status": "full",
      "receipt": {
        "appliedAt": "ISO-8601",
        "installed": [
          { "skillKey": "slides", "contentHash": "sha256..." }
        ]
      }
    }
  ],
  "skippedConflicts": [
    {
      "packId": "uuid",
      "skillKey": "slides",
      "reason": "same_name_exists"
    }
  ]
}
```

- `status`：`full` | `partial`（存在跳过时为 `partial`，UI 显示「有跳过」）。
- `receipt`：应用成功时记录实际写入的技能及其安装时哈希。仅用于 §5.4 的卸载确认；**不**驱动自动删除、不参与同步。
- **无** `managedSkills`、`baselineHash`、`contentHash` 托管表。
- 内容哈希仅用于：库技能目录摘要、技能包内引用校验、预检时「引用是否仍指向当前库内容」、以及 §6 的现算对比。

哈希：对技能目录按相对路径排序后的路径 + 字节内容计算；排除 `.DS_Store` 等 OS 元数据。配置写入采用临时文件 + 原子重命名。

## 8. API 边界

建议接口（路由名可随现有风格微调）：

| 接口 | 用途 | 交付阶段 |
| --- | --- | --- |
| `GET/PUT /api/skill-library` | 获取或设置技能库目录；返回库技能列表。 | MVP |
| `POST /api/skill-library/import` | 市场 / 本地 / Git 发现结果导入库。 | MVP |
| `GET/POST/PATCH/DELETE /api/skill-packs` | 技能包 CRUD。 | MVP |
| `POST /api/workspace-skill-packs/preview` | 按 `cwd` + 目标 packId 集合返回不可变预览计划。 | MVP |
| `POST /api/workspace-skill-packs/apply` | 按已确认预览执行复制与状态写入（**仅安装与改标签，不删技能目录**）。 | MVP |
| `DELETE /api/workspace-skill-packs/:packId` | 卸载技能包：用回执算删除列表 → 用户确认 → 删目录 → 去标签。 | v1.1 |
| `POST /api/skills/:skillKey/overwrite` | 项目↔库覆盖（body 指定方向 `toProject` / `toLibrary`）；另存改名（`saveAs`）。 | v1.2 |

- `apply` 须携带预览 ID 或输入版本；预检后目录变化则拒绝并要求重新预检。
- `overwrite` 为一次性原子替换，不建同步关联；成功返回 `{ok: true}`。
- **不提供** `POST /api/skills/sync`（无同步状态机，不需要）。

与现有接口关系：

- `/api/skills`、`/api/skills/install` 等继续服务「当前工作空间 / 全局」直接安装与列表。
- 获取技能进库走 skill-library 路径，避免与「装到当前项目」默认行为混淆。

## 9. 验收标准

- 用户能选择一个本地目录作为技能库，并看到其中 `.pi/skills` 下的技能。
- 用户能从市场搜索、本地目录、Git 将技能导入技能库（不写当前项目）。
- 用户能创建名为「PPT大师」的技能包，并从库中加入多个技能。
- **打开已有空项目**，应用两个技能包后，项目 `.pi/skills` 包含它们技能的去重并集。
- 项目已有同名技能时，应用不覆盖；确认框写明跳过原因；技能包标签仍可挂上并显示「有跳过」。
- 移除一个技能包后，项目 `.pi/skills` 内文件**全部保留**；仅标签与 `skill-packs.json` 更新。
- 两个技能包共同引用的技能，移除其中一个包后技能文件仍在（本就不应被删）。
- 用户手动修改过的项目技能，在移除相关技能包后仍在且内容不变。
- 技能库或技能包更新后，已应用项目中的技能文件不变；仅新的预检/应用使用用户已显式更新后的包引用。
- UI 与文档中技能场景不出现与 Tool Preset 混淆的裸词「Preset」。

## 10. 实施顺序

### MVP

1. 技能库配置、目录扫描、技能目录哈希、全局 `packs` 持久化。
2. 技能包 CRUD 与技能包模态框。
3. 工作空间预检 / 应用（只复制 + 写 `skill-packs.json`，含 `receipt`）与标签 UI（含「有跳过」）。
4. 市场 / 本地 / Git 导入接入技能库。
5. （可选增强）选择/创建工作空间流程上挂可选技能包。

**第 1–4 步构成主路径闭环**；第 5 步不挡 MVP。

### v1.1 — 卸载

6. 「卸载技能包…」入口与确认弹窗（§5.4、§12.1）。
7. `DELETE /api/workspace-skill-packs/:packId`。

### v1.2 — 库对照

8. 技能详情页「与库不同」提示（§6.1）。
9. 覆盖/另存三个操作（§6.2）与 `POST /api/skills/:skillKey/overwrite`。

### 不做

- 实时同步引擎、分叉状态机、三路合并、关标签自动删。

## 11. 与旧稿对照（决策摘要）

| 旧稿 | 共识后 |
| --- | --- |
| 名：Preset | **技能包 / Skill Pack** |
| 隐喻：复制 + 托管 + 同步 | **App Store 合集：安装 + 回执 + 一次性覆盖** |
| 移除 Preset → 条件删除托管技能 | **移除标签不动文件；卸载带确认勾选** |
| 推/拉/分叉同步（带基线状态机） | **现算哈希对比 + 三个一次性操作（覆盖/更新/另存）** |
| 托管自动删除 | **安装回执 + 卸载时现算引用** |
| `2/3 已应用` | **「有跳过」** |
| 主路径绑新建工作空间 | **主路径 = 当前工作空间应用** |
| `managedSkills` / baseline | **`receipt[]`（仅安装快照，不驱动同步）** |
| 项目状态 `preset-state.json` | **`skill-packs.json`** |

## 12. 进阶设计：卸载与库对照

> 本节是 §5.4（卸载）和 §6（库对照）的详细设计展开。不阻塞 MVP。

### 12.1 卸载技能包（v1.1）

MVP 已定「移除标签 ≠ 删文件」。卸载是用户**主动点的**带确认的清理动作。

#### 安装回执

每次 `apply` 成功后写回执（§7.2）：

```json
{
  "packId": "uuid",
  "status": "full",
  "receipt": {
    "appliedAt": "2025-01-01T00:00:00Z",
    "installed": [
      { "skillKey": "slides", "contentHash": "sha256-abc" },
      { "skillKey": "review", "contentHash": "sha256-def" }
    ]
  }
}
```

- 只记**实际写入**的技能（冲突跳过的不记）。
- 哈希为安装时源内容摘要，不随项目后续修改更新。
- 以后再执行 `apply`（增量或重装）时追加/更新回执中的对应条目。

#### 卸载弹窗

点「卸载技能包…」→ 服务端现算：

1. 读该包所有回执条目。
2. 对每条：检查当前项目目录是否存在 + 内容哈希。
3. 检查该 skillKey 是否也出现在其他已挂包的回执中。

确认框按三组展示：

| 分组 | 条件 | 默认 |
| --- | --- | --- |
| **将删除** | 文件仍在、哈希 = 回执、无其他包共用 | ✅ 勾选 |
| **已修改 · 将保留** | 文件仍在、哈希 ≠ 回执、无其他包共用 | ☐ 不勾选 |
| **被其他技能包共用 · 将保留** | 文件仍在、有其他包的回执也引用 | ☐ 不勾选，灰显 |

- 已不存在于磁盘的回执条目 → 不显示（自动忽略）。
- 用户可以手动勾选「已修改」和「共用」条目；不会阻止。

确认后删除勾选的目录 → 写回执标记本条已卸载（或直接移除 `appliedPacks` 条目）。

#### 更懒一档

如果 v1.1 只想最小成本交付：卸载弹窗 = 列出回执中所有 skillKey（仅名称，不算哈希），**默认全不选**，用户勾哪个删哪个。无分组，无共用检测。连「未改才默选」都不要。

**产品推荐**：上一档（三组+默认勾选）已经够简够好，再懒失去卸载便利性。

### 12.2 推/拉覆盖（v1.2）

在项目技能详情中，若同名库技能存在：

```text
[ 与库不同 ]  ← 纯提示，不是状态标签
[ 用库版本覆盖项目 ]  [ 用项目版本更新库 ]  [ 另存为新库技能… ]
```

两边哈希相同时按钮灰掉显示「与库相同」。

**安全约束：**

- 任何覆盖必须先弹确认，写明源/目标路径。
- 目标目录先写入临时位置再原子替换；失败不留半截目录。
- 「更新库」成功后不自动改技能包引用。技能包详情继续显示「引用需更新」直到用户显式更新。
- 「更新库」不自动创建/更新 git commit（技能库是否纳入版本管理由用户自行负责）。

### 12.3 分叉 → 另存（v1.2）

不设分叉状态。前端现算哈希不同时，「另存为新库技能…」就是处理分叉的方式（旧稿的「改名保留两份」）。

另存弹窗：

- 预填建议名（如 `{原 skillKey}-variant`）。
- 含「同时用这个版本创建新技能包」checkbox。
- 执行：项目目录 → 复制到 `<libraryRoot>/.pi/skills/{newSkillKey}`。

### 12.4 设计总则

> **标签**表达「我在用哪些场景」；**回执**记住「上次为这个场景装过什么」；**卸载**是带名单的后悔药；**与库不同**时用覆盖/另存两个锤子，绝不设同步引擎。

## 13. 实现前须再核实的事实（非产品争议）

- 技能库作为 `npx skills add` 的 cwd 时，是否稳定写入 `<libraryRoot>/.pi/skills`（与项目 scope 行为一致）；若 CLI 另有约定，导入实现须跟随 CLI，并在本文件补一行事实说明。
- 全局配置落盘位置（Pi Web 专用配置路径）与现有 settings / models 配置并列方式，实现时与 `AGENTS.md` 中 auth/models 配置风格对齐。
- 项目状态文件是否提交进用户 Git：默认写入 `.pi/skill-packs.json`；是否加入产品内「建议 gitignore」提示，可在实现时单独定，**不影响**应用语义。
